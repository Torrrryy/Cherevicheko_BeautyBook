# BeautyBook (vika)

Модуль онлайн-записи клиентов в салон красоты: клиенты, мастера, услуги и записи с проверкой доступности мастера.

## Стек

| Слой     | Технологии                                                      |
| -------- | --------------------------------------------------------------- |
| Frontend | Next.js 16 (App Router), React 19, TypeScript, Tailwind CSS 4   |
| API      | tRPC 11, React Query, Zod, SuperJSON                            |
| БД       | PostgreSQL, Drizzle ORM                                         |
| Auth     | Better Auth (email + пароль, роли `admin` / `staff` / `client`) |

## Требования

- Node.js 18+
- PostgreSQL (Docker или локально)
- npm

## Быстрый старт

```bash
cd vika
npm install
cp .env.example .env

docker compose up -d
npm run db:push
npm run db:seed

npm run dev
```

Приложение: [http://localhost:3001](http://localhost:3001)

## Переменные окружения

| Переменная            | Назначение                                                               |
| --------------------- | ------------------------------------------------------------------------ |
| `DATABASE_URL`        | PostgreSQL (`postgresql://postgres:postgres@localhost:5432/BeautySalon`) |
| `BETTER_AUTH_SECRET`  | Секрет сессий (≥ 32 символов в проде)                                    |
| `BETTER_AUTH_URL`     | `http://localhost:3001`                                                  |
| `NEXT_PUBLIC_APP_URL` | Публичный URL для клиента Better Auth                                    |

## Роли

| Роль     | Возможности                                       |
| -------- | ------------------------------------------------- |
| `admin`  | Справочники, все записи, календарь любого мастера |
| `staff`  | Своё расписание, отмена своих записей             |
| `client` | Свои записи, новая запись, история визитов        |

## API по ТЗ (tRPC)

| Операция                     | Процедура                       |
| ---------------------------- | ------------------------------- |
| Создать запись               | `appointments.create`           |
| Обновить запись              | `appointments.update`           |
| Отменить запись              | `appointments.cancel`           |
| Проверка доступности мастера | `staff.checkAvailability`       |
| Расписание на день           | `staff.scheduleByDay`           |
| Расписание на неделю         | `staff.scheduleByWeek`          |
| История клиента              | `clients.history`               |
| Предстоящие записи по услуге | `services.upcomingAppointments` |

## Тесты

```bash
# Unit-тесты (без PostgreSQL)
npm run test:unit

# Интеграционные API-тесты (нужна БД)
cp .env.test.example .env.test   # опционально
docker compose up -d
npm run db:push
npm run test:integration

# Все тесты
npm run test
```

Интеграционные тесты используют префикс `test-api` в email/именах и автоматически пропускаются (`describe.skipIf`), если `DATABASE_URL` недоступен.

Покрытие:

- **unit** — домен доступности (`availability`), Zod-схемы, утилиты
- **api** — tRPC + PostgreSQL: записи, мастера, клиенты, услуги, RBAC

## Оценка курсового проекта «BeautyBook»

**Проект:** модуль онлайн-записи клиентов салона красоты  
**Курс:** 3 курс бакалавриата  
**Дата оценки:** 2026-05-19  
**Эксперт:** SourceCraft Code Assistant Agent

### Общее впечатление

Проект представляет собой полнофункциональное веб-приложение для записи клиентов в салон красоты с ролевой моделью (`admin` / `staff` / `client`), аутентификацией и доменной логикой проверки доступности мастеров. Кодовая база организована по слоям, использован современный стек (Next.js App Router, tRPC 11, Drizzle ORM, Better Auth, Zod). Реализованы все ключевые операции из ТЗ: CRUD записей, расписание на день/неделю, история клиента, предстоящие записи по услуге. Работа соответствует уровню хорошей курсовой для 3 курса.

### Детальная оценка по критериям

#### 1. Сложность проекта (0–10)

**Оценка: 8**

**Обоснование:** Многостраничное приложение с аутентификацией (Better Auth + Drizzle adapter), PostgreSQL, 4 tRPC-роутера по доменам (`appointments`, `staff`, `clients`, `services`), три роли с RBAC, отдельный доменный модуль проверки пересечения временных слотов и рабочих часов салона, календарь на неделю, live-проверка доступности при записи.

**Недостатки:** нет real-time (subscriptions/WebSocket), напоминаний (email/SMS), оплаты, очереди ожидания; миграции Drizzle в репозитории не зафиксированы (используется `db:push`).

#### 2. Архитектура проекта (0–10)

**Оценка: 8**

**Обоснование:** Чёткое разделение: `app/` (страницы), `components/` (UI), `server/api/routers/` (tRPC по доменам), `server/domain/` (бизнес-логика доступности), `server/api/schemas/` (Zod), `server/api/lib/access.ts` (RBAC), `server/db/schema/` (Drizzle). Better Auth и подключение к БД вынесены в `lib/`.

**Недостатки:** нет dependency injection, домен `availability` напрямую зависит от Drizzle; граница «роутер ↔ домен» местами размыта (часть проверок в процедурах).

#### 3. Качество кода (0–10)

**Оценка: 8**

**Обоснование:** Строгий TypeScript, типобезопасный tRPC end-to-end, валидация входов через Zod, ошибки через `TRPCError` с понятными сообщениями на русском, осмысленные имена, компактные функции в `availability.ts` и `access.ts`, SuperJSON для дат.

**Недостатки:** встречаются повторяющиеся паттерны в страницах (формы без React Hook Form, как в эталонном FitFlow); нет exhaustive-проверок для статусов записи на уровне TypeScript.

#### 4. UI/UX (0–10)

**Оценка: 8**

**Обоснование:** Адаптивный layout с боковым меню по ролям, страницы для всех сценариев ТЗ, таблицы записей, недельный календарь, форма новой записи с проверкой слота, индикаторы загрузки и ошибок на формах, единая тема на Tailwind CSS 4.

**Недостатки:** нет skeleton-состояний и отдельного компонента пустого списка; слабая a11y (мало `aria-*`); Playwright в зависимостях, но e2e-сценарии не реализованы.

#### 5. Качество тестов (0–10)

**Оценка: 8**

**Обоснование:** Unit-тесты (Vitest) на домен доступности, Zod-схемы и утилиты; интеграционные API-тесты на tRPC с PostgreSQL (записи, мастера, клиенты, услуги, RBAC); быстрые тесты без БД на `UNAUTHORIZED` / `FORBIDDEN`; хелперы `fixtures`, `db`, `createCallerForRole`; интеграции корректно пропускаются при недоступной БД.

**Недостатки:** нет e2e (Playwright), нет отчёта покрытия (coverage), CI не настроен; интеграционные тесты требуют ручного поднятия PostgreSQL.

#### 6. Качество документации (0–10)

**Оценка: 8**

**Обоснование:** README с быстрым стартом, Docker Compose, переменными окружения, ролями, таблицей API по ТЗ, тестовыми учётками, структурой проекта и разделом про тесты; есть `.env.example` и `.env.test.example`.

**Недостатки:** нет диаграммы схемы БД, JSDoc для публичных функций, описания алгоритма пересечения интервалов; README короче, чем у соседнего эталонного проекта FitFlow.

### Рекомендации по улучшению

- **Сложность:** добавить уведомления о записи (email) или real-time обновление календаря через tRPC subscriptions.
- **Архитектура:** вынести чистую логику доступности в функции без Drizzle; подключать репозиторий через интерфейс или контекст.
- **Качество кода:** унифицировать формы через React Hook Form + Zod resolver; ввести enum-типы для `appointment.status` на уровне приложения.
- **UI/UX:** skeleton при загрузке списков, компонент пустого состояния, улучшить accessibility (aria-labels, фокус в модалках).
- **Тесты:** добавить Playwright smoke (вход → новая запись → отмена), настроить GitHub Actions, включить `vitest --coverage`.
- **Документация:** ER-диаграмма схемы Drizzle, комментарий к алгоритму `intervalsOverlap`, раздел «Типичные проблемы».

### Заключение

Проект заслуживает положительной оценки и соответствует требованиям курсовой работы по разработке веб-приложений. Студент продемонстрировал умение проектировать предметную область (салон красоты), реализовать типобезопасный API на tRPC, работать с PostgreSQL через Drizzle и покрыть критичную логику тестами. При доработке UI и CI проект может служить основой для production-MVP.
